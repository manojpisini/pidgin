protocol overview

Pidgin is a safety-gated agent-to-agent communication protocol. Every packet travels through a 7-stage pipeline. The protocol enforces a deny-list model with 10 safety rules.

packet format

@run <run_id>
wf=<workflow>         mode=draft|approval
in=[<inputs>]          out=[<outputs>]
do=[<allowed>]         deny=[<denied>]
risk=low|med|crit      human=yes|no
ttl=24                 note="text"

handoff fields

from=agent_alpha        to=agent_beta
chain=[alpha,beta,gamma]
handoff=forward|relay|delegate
reply_to=<run_id>      origin=<run_id>

handoff modes

  • forward pass packet unchanged
  • relay agent processes and re-wraps
  • delegate full responsibility transfer

context transfer

Large context is not embedded inline. Pidgin uses reference-based context transfer to keep packets small and fast.

context_file=[file:path/to/doc.md,file:other.md]
context_id=ctx_abc123    # pointer to uploaded context
context="inline text"   # small inline context
context_b64=SGVsbG8...   # base64 encoded

context methods

  • context_file — array of file references, resolved at runtime. Best for large docs, configs, code.
  • context_id — pointer to pre-uploaded context store. Agent fetches by ID before processing.
  • context — inline quoted string. Use for small annotations, notes, instructions.
  • context_b64 — base64 encoded binary. Use for images, serialized data, non-text payloads.

context size limits

  • Inline context: recommended max 4KB (parser handles more, but packets should stay fast)
  • context_file: no limit, files resolved from filesystem
  • context_id: no limit, depends on store backend
  • context_b64: recommended max 1MB inline, larger via context_id

directives

  • @run execute a workflow
  • @result outcome of a task
  • @approval human sign-off
  • @context request additional context

pipeline

01 LEXER

Tokenizes raw .pgn text. Detects directive. Handles encoding.

02 PARSER

Extracts key-value pairs. Parses arrays. Generates run_id.

03 SYNTAX

Validates required fields. Checks types and ranges.

04 SCHEMA

Workflow schema validation. Risk/human/mode checks.

05 SAFETY

Evaluates 10 safety rules. Blocks dangerous combos.

06 RESOLVE

Resolves reference aliases. Maps to full paths.

07 EXPAND

Generates YAML execution spec with routing.

safety rules

SG-1   deny contains actions from do
SG-2   critical risk with human=no
SG-3   publish/send in do
SG-4   delete in do
SG-5   secrets in do or deny
SG-6   file traversal in inputs
SG-7   missing human on critical
SG-8   invalid risk level
SG-9   approval mode without human=yes
SG-10  chain depth exceeded
