PIDGIN Docs

The Agent Handoff Protocol.

PIDGIN standardizes how AI agents hand off work with safety, clarity, and control.
Small packets. Strong guarantees. Built for production.

WHAT IS PIDGIN?

PIDGIN is a lightweight protocol for agent-to-agent handoffs. It defines a minimal packet format, strict safety rules, and a set of commands so agents can collaborate safely and predictably.

Docs  ›  Install

INSTALLATION

Prerequisites

PIDGIN requires Rust 1.75+ and Cargo. If you don't have Rust installed, use rustup:

curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

Install from crates.io

The fastest way to get PIDGIN. Installs the pgn CLI and the pidgin-lang library:

cargo install pidgin-lang

This installs the pgn binary to ~/.cargo/bin/. Make sure this is in your PATH.

Initialize a Workspace

Create a new PIDGIN project with the standard directory structure:

pgn init my-project
cd my-project

This creates:

  • .pidgin/config.yaml - Project configuration
  • .pidgin/safety_rules.yaml - Custom safety rules
  • .pidgin/packets/ - Directory for .pgn packet files
  • .pidgin/aliases.yaml - Reference alias definitions

Verify Installation

pgn --version
pgn docs

Install from Source

Clone the repository and build with Cargo:

git clone https://github.com/pidgin-protocol/pidgin.git
cd pidgin
cargo build --release
cargo install --path crates/pidgin-cli
Note: Building from source requires Rust 1.75+ and approximately 2 minutes on modern hardware.

Docs  ›  Quick Start

QUICK START

Get up and running with PIDGIN in under 5 minutes. This guide walks you through creating your first agent-to-agent handoff.

Step 1: Create a Packet

Create a file called hello.pgn in your project:

@run hello.world
wf=code_review
mode=draft
in=[main.rs]
out=[review]
do=[read,analyze,report]
deny=[publish,send,delete]
risk=low
human=no
from=orchestrator
to=executor_alpha

Step 2: Parse the Packet

Parse the raw .pgn text into structured fields:

pgn parse hello.pgn

Output shows the parsed directive, run ID, field count, and any parse errors.

Step 3: Validate

Check that all required fields are present and types are correct:

pgn validate hello.pgn

Step 4: Run Safety Checks

Execute all 9 safety rules against the packet:

pgn check hello.pgn

If all rules pass, the packet is safe to execute. If any rule blocks, you'll see which rule fired and why.

Step 5: Expand to Execution Plan

Convert the packet into a YAML execution spec:

pgn expand hello.pgn

The expanded YAML includes resolved aliases, injected metadata, and the execution route.

Step 6: Run the Workflow

pgn run hello.pgn

This executes the full 7-stage pipeline: lex, parse, syntax, schema, safety, resolve, expand. The result is logged to your audit trail.

Complete Example

# One-shot: parse, validate, check, expand
pgn check hello.pgn

# Full pipeline execution
pgn run hello.pgn

# View metrics from the last run
pgn measure --last
Tip: Use pgn docs to open this documentation in your terminal at any time.
Docs  ›  Packet Format

PACKET FORMAT

Pidgin packets are plain-text key-value documents with a directive header. Every packet begins with a directive (like @run) followed by a run ID, then zero or more key=value fields.

Anatomy

@directive run_id
key=value
key=[value1,value2]
key="quoted string"

Directive

The first line must start with @ followed by the directive name and a run identifier:

DirectivePurposeExample
@runExecute a workflow@run code.review
@resultReport task outcome@result review.complete
@approvalRequest human sign-off@approval deploy.prod
@contextRequest context transfer@context design.spec

Required Fields

FieldTypeDescription
wfstringWorkflow identifier. Maps to a workflow definition in your config.
modedraft | approvaldraft = auto-execute. approval = requires human sign-off.
inarrayInput resources. Supports file: prefix for file references.
outarrayExpected output identifiers.

Optional Fields

FieldTypeDefaultDescription
doarray[]Allowed actions for this packet.
denyarray[]Denied actions. Checked by SG-5.
risklow | med | critlowRisk level. Affects safety rules.
humanyes | nonoWhether human oversight is required.
ttlinteger24Time-to-live in hours.
notestring-Free-text annotation.
vstring1.0.0Protocol version.

Handoff Fields

FieldTypeDescription
fromstringSource agent identifier.
tostringDestination agent identifier.
chainarrayOrdered list of agents in the handoff chain.
handoffforward | relay | delegateHandoff mode. forward = pass unchanged, relay = process and re-wrap, delegate = full transfer.
reply_torun_idReference to the originating run.
originrun_idOriginal run ID at chain start.

Context Transfer

FieldTypeLimitDescription
context_filearrayUnlimitedFile references using file: prefix.
context_idstringUnlimitedPointer to a context store.
contextstring4KBInline context text.
context_b64string1MBBase64-encoded binary context.

Field Naming

Field names must be alphanumeric with underscores. Reserved words: directive, run_id, timestamp, signature. Field values support these formats:

  • Plain strings: wf=code_review
  • Arrays: in=[file:main.rs,file:lib.rs]
  • Quoted strings: note="needs review by Friday"
  • Integers: ttl=48
  • Booleans: human=yes
Docs  ›  Commands

CLI COMMANDS

The pgn command-line interface provides tools for parsing, validating, checking, and executing PIDGIN packets.

pgn init

Initialize a new PIDGIN workspace with default configuration files.

pgn init [project-name]

Creates .pidgin/ directory with config, safety rules, aliases, and a packets directory.

pgn parse

Parse raw .pgn text into structured fields. Reports directive, run ID, field count, and any syntax errors.

pgn parse packet.pgn
pgn parse --json packet.pgn    # JSON output

pgn validate

Validate that all required fields are present and types match expectations. Returns a list of validation errors (empty if valid).

pgn validate packet.pgn
pgn validate --strict packet.pgn   # fail on warnings

pgn check

Run all safety rules (SG-1 through SG-9) against the packet. Returns pass/fail for each rule and overall verdict.

pgn check packet.pgn
pgn check --rules SG-1,SG-2 packet.pgn   # specific rules

pgn resolve

Resolve reference aliases, inject metadata, and prepare the packet for expansion.

pgn resolve packet.pgn

pgn expand

Expand the packet into a complete YAML execution specification. Includes resolved routes, metadata, and timestamps.

pgn expand packet.pgn
pgn expand --output plan.json packet.pgn   # save to file

pgn run

Execute the full 7-stage pipeline: lex, parse, syntax, schema, safety, resolve, expand. Logs results to the audit trail.

pgn run packet.pgn
pgn run --dry-run packet.pgn   # simulate only

pgn measure

Collect and display metrics from packet processing. Shows latency, throughput, error rates, and safety rule statistics.

pgn measure --last          # metrics from last run
pgn measure --window 1h     # metrics from last hour
pgn measure --output report.json

pgn docs

Open the PIDGIN documentation in your default browser.

pgn docs

pgn serve

Start the PIDGIN HTTP server with the monitoring dashboard.

pgn serve --bind 0.0.0.0:3847

Global Options

OptionDescription
--jsonOutput results as JSON
--quietSuppress non-error output
--verboseEnable detailed logging
--config <path>Use custom config file
Docs  ›  Safety Rules

SAFETY RULES (SG-1 through SG-9)

PIDGIN enforces 9 safety rules on every packet before execution. No rule failure is allowed to pass. Rules are evaluated in order; the first failure blocks the packet.

Critical: Safety rules cannot be disabled or bypassed. They are hard-coded in the Rust core and evaluated on every pipeline run.
SG-1Schema Validation

Validates that the packet conforms to the expected schema. Checks field names, types, array formatting, and value ranges. Rejects packets with unknown fields in strict mode.

SG-2Signature Verification

Verifies the packet signature against the signing key. Ensures the packet has not been tampered with in transit. Required for mode=approval packets.

SG-3Workflow Allowlist

Checks that the wf field matches a known workflow in the project configuration. Blocks packets referencing undefined workflows.

SG-4Mode Constraints

Enforces mode-specific rules: approval mode requires human=yes. draft mode cannot contain destructive actions in do.

SG-5Denylist Enforcement

Checks that no action in the deny list also appears in the do list. If an action is both allowed and denied, the packet is blocked.

SG-6Risk Thresholds

Enforces risk-level constraints: crit risk requires human=yes and mode=approval. med risk requires explicit do list.

SG-7Human Control

Ensures human oversight for high-risk operations. Packets with risk=crit or destructive actions must have human=yes.

SG-8Output Constraints

Validates that output identifiers in out are non-empty and don't reference restricted resources. Prevents data exfiltration.

SG-9Audit Logging

Ensures every packet generates an audit trail entry. Blocks packets that attempt to suppress logging or bypass the audit system.

Rule Evaluation Order

Rules are evaluated in numerical order (SG-1 through SG-9). This order is intentional:

  1. SG-1 first: Schema validation must pass before any semantic checks.
  2. SG-2 second: Signature must be valid before checking content.
  3. SG-3..SG-8:
  4. SG-9 last: Audit logging is checked after all content rules.

Custom Rules

Add custom rules in .pidgin/safety_rules.yaml:

custom_rules:
  - id: CR-1
    name: "No production secrets"
    check: "deny contains 'secrets' or 'credentials'"
    severity: high
    action: block
Docs  ›  Multi-Agent Setup

MULTI-AGENT SETUP

PIDGIN supports complex multi-agent workflows where packets are handed off between agents in a chain. Each agent has its own identity, capabilities, and safety profile.

Agent Configuration

Define agents in .pidgin/config.yaml:

agents:
  orchestrator:
    role: orchestrator
    model: claude-3.5-sonnet
    capabilities: [parse, route, delegate, context_resolve]
    max_chain_depth: 5

  executor_alpha:
    role: executor
    model: gpt-4o
    capabilities: [code_review, draft, expand, file_read]
    trusted_by: [orchestrator]

  executor_beta:
    role: executor
    model: claude-3-haiku
    capabilities: [validate, check, resolve]
    trusted_by: [orchestrator]

  human_approver:
    role: approver
    model: null
    capabilities: [approve, reject, override]

Handoff Modes

ModeDescriptionUse Case
forwardPass packet unchanged to next agentSimple routing, no processing
relayAgent processes and re-wraps the packetIntermediate processing steps
delegateFull transfer of responsibilityAgent takes ownership of the task

Chain Depth

The chain field tracks the path of agents the packet has visited. SG-10 (chain depth exceeded) fires when the chain exceeds the configured maximum (default: 5). This prevents infinite loops in agent handoffs.

chain=[orchestrator,executor_alpha,executor_beta]
handoff=relay
origin=code.review.001

Trust Relationships

Agents can only receive packets from agents listed in their trusted_by field. This prevents unauthorized agents from injecting packets into a workflow.

Docs  ›  LangGraph

LANGGRAPH INTEGRATION

PIDGIN integrates with LangGraph to provide safety-gated agent handoffs in LangChain-based workflows.

Setup

pip install pidgin-langchain

Usage

from pidgin_langchain import PidginNode

# Add PIDGIN safety checks to your LangGraph workflow
graph = StateGraph(AgentState)
graph.add_node("pidgin_check", PidginNode(
    rules=["SG-1", "SG-2", "SG-5", "SG-6"],
    on_block="reject"
))
graph.add_edge("llm_call", "pidgin_check")
graph.add_conditional_edges("pidgin_check", route_by_safety)

Configuration

The LangGraph integration reads your .pidgin/config.yaml automatically. You can override settings per-node:

PidginNode(
    config_path=".pidgin/config.yaml",
    rules=["SG-1", "SG-2"],        # subset of rules
    strict=False,                    # allow warnings
    timeout_ms=5000                  # per-check timeout
)
Docs  ›  CrewAI

CREWAI INTEGRATION

PIDGIN provides safety-gated handoffs for CrewAI multi-agent crews. Each task handoff passes through the full PIDGIN pipeline.

Setup

pip install pidgin-crewai

Usage

from pidgin_crewai import PidginCrew

crew = PidginCrew(
    agents=[researcher, writer, reviewer],
    safety_rules=["SG-1", "SG-5", "SG-7"],
    on_block="log_and_continue"
)

result = crew.kickoff(inputs={"topic": "AI safety"})

How It Works

  • Each agent task generates a .pgn packet before execution.
  • The packet is validated through the PIDGIN pipeline.
  • If safety rules pass, the task executes normally.
  • If a rule blocks, the crew logs the block and can either halt or skip the task.
Docs  ›  A2A Integration

A2A (AGENT-TO-AGENT) INTEGRATION

PIDGIN implements the A2A protocol for standardized agent-to-agent communication across different frameworks and platforms.

Protocol

A2A uses PIDGIN packets as the wire format. Any agent that speaks PIDGIN can communicate with any other PIDGIN-speaking agent, regardless of the underlying framework.

Endpoint

POST /a2a/handoff
Content-Type: text/plain

@run agent.handoff
wf=collaborative_task
mode=draft
from=agent_alpha
to=agent_beta
in=[task_data]
out=[result]

Discovery

Agents announce their capabilities via the /a2a/capabilities endpoint:

GET /a2a/capabilities
{
  "agent": "executor_alpha",
  "capabilities": ["code_review", "draft", "expand"],
  "accepted_directives": ["@run", "@result"],
  "max_chain_depth": 5
}

HOST CONFIGURATION

PIDGIN reads configuration from .pidgin/config.yaml in your project root. This file controls agent definitions, safety rules, workflow mappings, and runtime behavior.

Full Config Reference

# .pidgin/config.yaml
version: "0.4.0"

server:
  host: "0.0.0.0"
  port: 3847
  metrics_enabled: true

agents:
  default:
    role: executor
    model: gpt-4o
    max_chain_depth: 5
    capabilities: []

safety:
  rules:
    - SG-1
    - SG-2
    - SG-3
    - SG-4
    - SG-5
    - SG-6
    - SG-7
    - SG-8
    - SG-9
  custom_rules_path: ".pidgin/safety_rules.yaml"
  strict_mode: false

workflows:
  code_review:
    description: "Code review workflow"
    required_fields: [wf, mode, in, out]
    optional_fields: [do, deny, risk, human]

logging:
  level: info
  audit_trail: true
  output_path: ".pidgin/logs/"

aliases:
  path: ".pidgin/aliases.yaml"
  auto_resolve: true

Docs  ›  Rust Library

RUST LIBRARY

The pidgin-lang crate provides the core protocol implementation in Rust. Use it to build custom tools, integrations, or embed PIDGIN in your own applications.

Add Dependency

[dependencies]
pidgin-lang = "0.4.0"

Basic Usage

use pidgin_lang::{parse, validate, check, expand};

fn main() {
    let packet = "@run code.review\nwf=generic_review\nmode=draft\nin=[main.rs]\nout=[review]";

    // Parse
    let parsed = parse(packet).expect("parse failed");
    println!("Directive: {}", parsed.directive);
    println!("Fields: {}", parsed.fields.len());

    // Validate
    let errors = validate(&parsed);
    if !errors.is_empty() {
        eprintln!("Validation failed: {:?}", errors);
        return;
    }

    // Safety check
    let safety = check(&parsed);
    if safety.blocked {
        eprintln!("Blocked by: {:?}", safety.fired_rules);
        return;
    }

    // Expand
    let plan = expand(&parsed).expect("expand failed");
    println!("YAML:\n{}", plan.yaml);
}

API Reference

FunctionSignatureReturns
parsefn parse(input: &str) -> Result<Packet>Parsed packet with directive, run_id, fields
validatefn validate(packet: &Packet) -> Vec<Error>List of validation errors (empty = valid)
checkfn check(packet: &Packet) -> SafetyResultSafety verdict with fired rules
expandfn expand(packet: &Packet) -> Result<Plan>YAML execution spec with route
resolvefn resolve(packet: &Packet, aliases: &Aliases) -> PacketPacket with resolved aliases

API REFERENCE

The PIDGIN HTTP server exposes a REST API for packet processing. All endpoints accept raw .pgn text and return JSON responses.

Base URL

http://localhost:3847/api/v1

Endpoints

MethodPathDescription
POST/parseParse raw .pgn text into structured fields
POST/validateValidate packet fields and types
POST/checkRun safety rules against packet
POST/expandExpand packet into YAML execution plan
POST/runExecute full pipeline (parse through expand)
GET/healthHealth check endpoint

POST /api/v1/parse

curl -X POST http://localhost:3847/api/v1/parse \
  -H "Content-Type: text/plain" \
  -d "@run test
wf=generic_review
mode=draft
in=[main.rs]
out=[review]"

Response Format

{
  "ok": true,
  "data": {
    "directive": "run",
    "run_id": "test",
    "field_count": 4,
    "fields": { "wf": "generic_review", "mode": "draft", ... }
  }
}

POST /api/v1/check

Returns safety rule evaluation results:

{
  "ok": true,
  "data": {
    "safety": {
      "blocked": false,
      "fired_rules": ["SG-1", "SG-3", "SG-5"],
      "passed_rules": ["SG-2", "SG-4", "SG-6", "SG-7", "SG-8", "SG-9"]
    },
    "resolution": [
      { "original": "agent_alpha", "resolved": "agent_alpha", "status": "RESOLVED" }
    ]
  }
}

Error Responses

{
  "ok": false,
  "error": "SG-5 blocked: action 'publish' in both do and deny lists"
}
Note: The API is designed for machine-to-machine communication. For human interaction, use the CLI or the dashboard.
