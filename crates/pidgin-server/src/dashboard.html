protocol overview

Pidgin is a safety-gated agent-to-agent communication protocol. Every packet travels through a 7-stage pipeline: lexing, parsing, syntax validation, schema checks, safety rule evaluation, reference resolution, and YAML expansion.

The protocol enforces a deny-list model: agents declare what they will do and what they will deny. Safety rules block dangerous combinations before execution.

Packets are 9-line text blocks in .pgn format. The first line is the directive. Fields are key=value pairs, one per line.

packet format

@run <run_id>
wf=<workflow>
mode=draft|approval
in=[<inputs>]
out=[<outputs>]
do=[<allowed_actions>]
deny=[<denied_actions>]
risk=low|med|crit
human=yes|no

required fields

  • wf — workflow identifier (e.g. generic_review, generic_draft_and_distribute)
  • mode — draft (auto) or approval (requires human gate)
  • in — array of input references (files, data, resources)
  • out — array of output references

optional fields

  • do — allowed actions array (e.g. [read, draft, review])
  • deny — denied actions array (e.g. [publish, send, delete])
  • risk — risk level: low, med, crit
  • human — yes (human-in-the-loop) or no (autonomous)

directives

  • @run — execute a workflow (most common)
  • @log — log an event without execution
  • @resolve — resolve reference aliases only
  • @emit — emit a signal to another agent

pipeline stages

01 — LEXER

Tokenizes raw .pgn text into line-based tokens. Detects the directive type from the first line. Handles whitespace normalization, comment stripping, and encoding detection.

Output: token stream with line references.

02 — PARSER

Extracts key-value pairs from tokens. Parses array syntax [a,b,c] into structured lists. Generates a deterministic run_id from the directive if not provided.

Output: field map with typed values.

03 — SYNTAX

Validates required fields are present. Checks field types (string vs array vs enum). Verifies value ranges (risk must be low/med/crit, mode must be draft/approval).

Output: validation result (pass/fail + error list).

04 — SCHEMA

Validates against workflow schema. Checks that the workflow identifier exists. Verifies that the requested actions are compatible with the risk level and human flag.

Output: schema compliance result.

05 — SAFETY

Evaluates 9 safety rules (SG-1 through SG-9). Rules check for dangerous combinations: actions that conflict, missing human gates on critical operations, path traversal attempts, and more.

Output: safety verdict (pass/block) + list of fired rules + human_required flag.

06 — RESOLVE

Resolves reference aliases to concrete paths. Maps short names to full filesystem or network paths. Checks reference existence and accessibility.

Output: resolution map (original → resolved path + status).

07 — EXPAND

Generates full YAML execution specification. Includes routing information, TTL, executor recommendation, and all resolved references. This is the final output delivered to the receiving agent.

Output: complete YAML document.

safety rules

SG-1  deny contains actions from do (conflict)
SG-2  critical risk with human=no (autonomous crit)
SG-3  publish or send in do (external distribution)
SG-4  delete in do (destructive action)
SG-5  secrets in do or deny (sensitive data)
SG-6  file traversal in inputs (path escape)
SG-7  missing human on critical (approval required)
SG-8  invalid risk level (schema violation)
SG-9  approval mode without human=yes (gate mismatch)

Rules fire independently. A packet can trigger multiple rules. Fired rules are logged and included in the safety output. Blocked packets never reach the expand stage.

api endpoints

POST /api/v1/parse      parse directive, extract fields
POST /api/v1/validate   check required fields
POST /api/v1/check      safety + resolution combined
POST /api/v1/expand     generate full YAML
POST /api/v1/run        execute full pipeline
GET  /api/v1/health     server status + host info

All POST endpoints accept text/plain body (raw .pgn text). Responses are JSON with ok boolean + data or error.

cli commands

pgn init                create .pidgin/ config directory
pgn parse <file>        parse .pgn packet
pgn validate <file>     validate required fields
pgn check <file>        safety rules + reference resolution
pgn expand <file>       expand to full YAML
pgn run <file>          execute complete pipeline
pgn serve               start HTTP server (default :3847)
pgn docs                display protocol documentation
